# Million List Test Task

Тестовое задание на позицию Fullstack Developer.

## Описание

Приложение реализует два связанных списка:

- **Available** — все элементы, кроме выбранных
- **Selected** — выбранные элементы в текущем порядке

Общий размер базового набора — **1 000 000 ID** (`1..1_000_000`).
Дополнительно можно вручную добавлять новые ID больше базового диапазона.

## Что реализовано

### Основной функционал

- два списка: `Available` и `Selected`
- фильтрация по ID в обоих списках
- infinite scroll порциями по 20 элементов
- добавление новых ID вручную
- перенос элемента из `Available` в `Selected`
- удаление элемента из `Selected`
- drag & drop сортировка в `Selected`
- сохранение выбора и порядка между reload страницы
- reset in-memory state для демонстрации

### Ограничение reorder в фильтре

Сортировка в `Selected` работает **только при пустом фильтре справа**.

Причина: при активном фильтре reorder видимого подмножества элементов создаёт неоднозначность относительно полного глобального порядка.

Поэтому принято сознательное решение:

- фильтр справа используется для поиска и просмотра
- drag & drop в режиме фильтра отключён
- глобальный порядок меняется только при работе с полным списком

Это делает поведение предсказуемым и упрощает архитектуру решения.

## Архитектура

### Backend — источник истины

Сервер хранит всё состояние в памяти процесса:

- базовый диапазон ID
- вручную добавленные ID
- выбранные ID
- порядок выбранных ID
- очереди изменений

### Frontend — отображение и действия

Frontend:

- отображает текущее состояние
- отправляет действия пользователя
- синхронизируется с backend
- не хранит источник истины у себя

## Серверное состояние

На backend хранится:

- `baseRange`
- `addedIds`
- `selectedIds`
- `selectedOrder`
- `version`
- `resetCount`
- `bootId`

### Что означают служебные поля

- `version` — растёт при обычных изменениях состояния
- `resetCount` — увеличивается при reset, даже если `version` снова стал `1`
- `bootId` — уникальный id запуска backend процесса

Это позволяет frontend различать:

- обычные изменения состояния
- reset
- новый цикл после рестарта сервера

## Очереди и batching

По условию задания запросы реализованы через очереди.

### `addQueue`

Используется для добавления новых ID.

- дедупликация по ID
- flush раз в **10 секунд**

### `mutateQueue`

Используется для изменений выбора и порядка:

- `select`
- `unselect`
- `reorder`

- дедупликация
- flush раз в **1 секунду**

## Почему не создаётся массив из миллиона элементов

Базовый диапазон `1..1_000_000` хранится как **виртуальный диапазон**, а не как заранее созданный массив.

Это позволяет:

- не держать миллион элементов в памяти
- не сериализовать миллион объектов
- отдавать только нужную страницу через `offset + limit`

## API

### System / sync

- `GET /api/health`
- `GET /api/state`
- `POST /api/debug/reset`

### Items

- `GET /api/items/available`
- `GET /api/items/selected`
- `GET /api/items/selected-all`
- `POST /api/items/add`
- `POST /api/items/select`
- `POST /api/items/unselect`
- `POST /api/items/reorder`

## Infinite scroll

Frontend хранит локальное состояние scroll/paging:

- `offset`
- текущую открытую глубину списка

Backend ничего не хранит про состояние прокрутки интерфейса.
Он только отдаёт нужный срез данных.

### Важный момент для `Available`

После server-side изменений список `Available` не схлопывается к первой странице, а перечитывается с сохранением уже открытой глубины.

### Новый серверный цикл

При:

- reset
- рестарте backend
- новой инициализации серверного state

Frontend сбрасывает глубину `Available` обратно к первой странице (20 элементов).

## Синхронизация frontend ↔ backend

Frontend:

- при загрузке страницы делает начальную синхронизацию
- затем периодически запрашивает `/api/state`
- сравнивает `bootId`, `resetCount`, `version`
- при необходимости перечитывает списки

### Почему одной `version` недостаточно

Если сервер перезапущен, `version` может снова стать `1`.
Поэтому для корректного обнаружения нового серверного цикла используются:

- `bootId`
- `resetCount`
- `version`

## Drag & Drop

Для reorder используется:

- drag & drop только в `Selected`
- backend endpoint `POST /api/items/reorder`
- полный текущий порядок `selected` читается через `GET /api/items/selected-all`

## Валидация add

### На frontend

- ID должен быть положительным целым числом
- ID из диапазона `1..1_000_000` нельзя добавить повторно

### На backend

Проверяется, что ID:

- не находится в базовом диапазоне
- не был уже добавлен
- не находится уже в очереди `add`

## Сохранение между reload страницы

По условию задания:

- выбор и порядок сохраняются между обновлениями страницы
- строка поиска сохраняться не должна

Это реализовано так:

- состояние хранится на backend
- после reload frontend заново запрашивает серверное состояние
- поля поиска остаются локальными и сбрасываются

## Reset

Для удобства проверки добавлена кнопка `Reset`.

Она:

- очищает серверное in-memory state
- очищает очереди
- не требует ручного перезапуска backend

После reset frontend сам определяет новый серверный цикл и возвращает интерфейс в начальное состояние.

## Проверка

### Базовый сценарий

1. Открыть приложение
2. Выбрать несколько элементов
3. Убедиться, что они исчезают слева и появляются справа
4. Изменить порядок справа
5. Обновить страницу
6. Проверить, что выбранные элементы и порядок сохранились

### Проверка add

1. Ввести ID больше `1_000_000`
2. Нажать `Add`
3. Дождаться flush
4. Убедиться, что элемент появился в `Available`

### Проверка remove

1. Удалить элемент из `Selected`
2. Дождаться flush
3. Убедиться, что он вернулся в `Available`

### Проверка infinite scroll

1. Прокрутить `Available`
2. Прокрутить `Selected`
3. Убедиться, что догрузка идёт порциями по 20

### Проверка reset

1. Добавить элементы / выбрать / отсортировать
2. Нажать `Reset`
3. Дождаться синхронизации
4. Убедиться, что приложение вернулось к начальному состоянию
